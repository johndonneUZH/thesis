\chapter{Experiments}\label{chap:experiments}
\section{Setup}
Having introduced the theoretical foundations of our correlation detection strategy, 
we now evaluate its effectiveness through a series of empirical experiments.

The data used from this point onward, namely the vector embeddings and their associated metadata, are described in detail in Section~\ref{sec:dataset} and Section~\ref{sec:correlated-attribute-generation}, respectively.

\section{Preliminary Experiments}
We conduct a controlled study on a synthetic dataset to analyze the behavior of filtered search. 
We consider four target selectivities:
\[
0.1\%,\; 1\%,\; 10\%,\; 50\%,
\]
which determine the width of the metadata query range and, consequently, the fraction of database vectors satisfying the predicate.

For each selectivity and each query vector, we generate three metadata intervals as described in Section~\ref{sec:query-generation}. 
This yields three workloads of query--predicate pairs:
\[
(\mathbf{q}, P^+),\qquad (\mathbf{q}, P^-),\qquad (\mathbf{q}, P^{\mathrm{rand}}),
\]
corresponding to the positive, negative, and random correlation regimes. 
For a query set of 10{,}000 vectors, this results in 30{,}000 query--predicate pairs in total.

For each workload, we compute the exact filtered ground truth before evaluating any approximate method.

Each query consists of a query vector together with a metadata interval \([l,u]\) that defines the filter condition. 
To avoid repeating identical work, queries with the same interval are processed together. 
For every distinct interval, we use the metadata values sorted in ascending order to locate, via binary search, the contiguous range of vector ids whose attributes fall inside \([l,u]\). 
This yields the exact admissible candidate set for that filter.

We then perform exact \(k\)-nearest neighbor search using a FAISS \texttt{IndexFlatL2}, while restricting the search space to this candidate set through an \texttt{IDSelector}. 
The resulting neighbors are used as the reference ground truth when measuring the recall of approximate methods.

Across all selectivities and correlation regimes, we benchmark three execution strategies, deeply covered in Chapter~\ref{subsec:implementation-details}.
We measure total runtime, queries per second (QPS), and recall@k with respect to the exact filtered ground truth. 

Results are reported in tabular form and visualized using summary plots.

\section{Correlation Analysis}\label{sec:correlation-analysis}

Alongside benchmarking, we evaluate whether the proposed metrics (Section~\ref{subsec:final-scores}) effectively capture correlation structure. We randomly sample a small subset of queries (10 in the current implementation, using a fixed random seed) and consider the three predicates associated with each query: positive, negative, and random.

For each query--predicate pair, we compute the true correlation as defined in Section~\ref{sec:correlation}, together with the heuristic scores \emph{heuristic}, \emph{tanh}, and \emph{p-gap}. These scores are then compared against the true correlation values.

The visualization consists of a three-panel scatter plot. In each panel, the x-axis represents the true correlation, while the y-axis represents one of the heuristic scores. Points are colored according to whether they originate from a positive, negative, or random query range. Each panel also includes the diagonal reference line \(y=x\), the zero axes, and the Pearson correlation coefficient between the heuristic score and the true correlation.

This allows us to assess both sign agreement and calibration quality of the surrogate metrics.

Finally, the data obtained in this step serves as input to the later \(\tau\)-estimation stage (Section~\ref{sec:tau-estimation}), where we determine which metric is the most suitable candidate for heuristic design.

Results are reported using summary plots.

\paragraph{Why are some queries misclassified by the heuristic metric?}

The heuristic is only a cluster-level approximation, whereas the true correlation is computed from the exact top-\(k\) nearest-neighbor rankings under filtering. The two quantities therefore measure related, but not identical, effects.

Its magnitude is determined by the Jensen--Shannon term, while its sign depends on whether the predicate mass inside the assigned cluster (\(p_c\)) is larger or smaller than the global mass (\(p_g\)). Both quantities rely on aggregate cluster statistics rather than on the actual admissible neighbors.

As a result, a query can be positive under the true metric but receive a negative heuristic score when the filter improves the exact neighborhood, yet the assigned cluster underrepresents the predicate.

For example, Query 3 in Figure~\ref{fig:corr-10pct} has \texttt{true\_correlation} \(= 0.0228\), but \texttt{heuristic\_correlation} \(= -0.2573\), since \(p_c = 0.00738 < p_g = 0.00984\).

Although the heuristic is imperfect, computing the true correlation at runtime would require exact filtered nearest-neighbor evaluation at query time, which is impractical. We therefore accept a limited risk of misclassification in exchange for a lightweight runtime proxy.

\paragraph{Why do tanh and pgap squish everything to 0 as we lower selectivity?}\label{par:shrink-toward-zero}
As selectivity decreases, the interval \([l,u]\) becomes narrower and covers a smaller portion of the attribute domain. Consequently, the probability mass captured by the interval decreases for both the local cluster distribution and the global distribution. Writing the interval width as \(\Delta\), and considering a small range around value \(a\), we obtain the local approximations

\[
p_c \approx f_c(a)\,\Delta,
\qquad
p_g \approx f_g(a)\,\Delta,
\]

where \(f_c\) and \(f_g\) denote the corresponding densities. As \(\Delta\) shrinks, both \(p_c\) and \(p_g\) decrease.

This directly explains the behavior of the \(p\)-gap score,

\[
\mathrm{pgap} = \mathrm{JS}\,(p_c - p_g),
\]

since

\[
p_c - p_g \approx \bigl(f_c(a)-f_g(a)\bigr)\Delta.
\]

Hence, the difference term contracts approximately linearly with the interval width, causing the score magnitude to decrease.

For the tanh-based score,

\[
\mathrm{tanh} = \mathrm{JS}\,\tanh\!\left(\log\frac{p_c}{p_g}\right),
\]

the mechanism is slightly different. A narrower interval does not automatically imply that \(p_c/p_g \to 1\); if the local density remains substantially larger or smaller than the global density, the ratio may still deviate from one. In practice, however, \(p_c\) and \(p_g\) are estimated from precomputed histogram bins rather than exact pointwise densities. Therefore, very small intervals often inherit nearly the same coarse bin mass in both distributions, which pulls the ratio toward one and drives

\[
\log\!\left(\frac{p_c}{p_g}\right) \to 0.
\]

As a result, the tanh score also approaches zero for highly selective ranges.

A representative example is shown below for the same negatively correlated query across selectivities:

\[
\begin{array}{c|c|c|c|c}
\text{Selectivity} & p_c & p_g & \text{pgap} & \text{tanh} \\
\hline
0.5   & 0.2320 & 0.4862 & -0.0900   & -0.2228 \\
0.1   & 0.1500 & 0.2033 & -0.0189   & -0.1045 \\
0.01  & 0.0925 & 0.0978 & -0.00188  & -0.0197 \\
0.001 & 0.0550 & 0.0556 & -0.000208 & -0.00376
\end{array}
\]

As selectivity decreases, both masses become smaller, their difference vanishes, and their ratio moves closer to one. Consequently, both \(p\)-gap and tanh scores collapse toward zero.
\section{Estimating \texorpdfstring{$\tau(s)$}{tau(s)}}
\label{sec:tau-estimation}

We estimate the selectivity-dependent threshold function $\tau(s)$ separately for each proposed correlation metric, following the null-calibration procedure described in Section~\ref{sec:tau}. For each selectivity level $s$, we generate a null workload in which the filter is independent of the query neighborhood, compute the corresponding correlation scores, and set $\tau(s)$ to the $99$th percentile of the absolute null scores.

Each calibration plot contains two panels. The left panel shows the histogram of absolute null correlation scores, i.e., the range of score magnitudes that can arise by chance under independence. The right panel shows the empirical cumulative distribution function (CDF) of the same scores. The red dashed line marks the $Q_{0.99}$ threshold used as $\tau(s)$.

Figure~\ref{fig:tau-heuristic} shows the calibration result for the heuristic metric across the considered selectivity levels. The resulting threshold is nearly invariant across selectivities. This is expected because the heuristic score magnitude is determined mainly by the Jensen--Shannon distance between the query cluster and the global attribute distribution. Selectivity affects the local lift, but the heuristic uses only the sign of this lift, not its magnitude.

\paragraph{Planning on the boundary.}
A further difficulty arises when integrating the threshold into the query planner, since rule-based routing at the threshold boundary induces discrete plan changes. 
As a result, queries whose metric values lie close to the threshold may be assigned to different execution plans, despite being nearly indistinguishable in practice. 
For example, depending on the planner policy, a query slightly above the threshold may be routed to PRE-FILTER, whereas one slightly below it may be routed to ACORN.

This highlights a limitation of hard thresholding: $\tau(s)$ should be understood as a practical routing threshold rather than a perfect semantic separator. 
 
One possible mitigation would be to use probabilistic routing for queries near the threshold. Instead of deterministically selecting a single plan, the system could choose between two candidate plans according to probabilities derived from a Gaussian distribution centered at the threshold. 
Queries far from the threshold would still be routed almost deterministically, while those close to it would allow smoother transitions between plans.

We do not explore this direction further in the present thesis, but it represents an interesting direction for future work.


\section{Planner Design}

We now construct a simple query planner that incorporates the estimated
correlation structure between the query and filter predicate, as determined by
the heuristic metric and its calibrated threshold.

Using the empirical results in Table~\ref{tab:performance-comparison}, together
with the correlation analysis from Section~\ref{sec:correlation-analysis}, we
identify which execution strategy performs best under different combinations of
selectivity and correlation, and route queries accordingly.

For highly selective predicates (\(s < 1\%\)), PRE-FILTER is consistently
preferred, as it achieves exact recall with very low latency.

For less selective predicates (\(s > 10\%\)), POST-FILTER performs well in
positively correlated and approximately random regimes, but deteriorates
substantially under negative correlation. In contrast, ACORN is more robust and
consistent across different correlated settings, although its advantage
diminishes as selectivity increases.

For the intermediate range (\(1\% \leq s \leq 10\%\)), we adopt two simplifying
assumptions. First, we assume that the efficiency of PRE-FILTER gradually
decreases as selectivity increases, with a crossover with the other two methods
at approximately \(5\%\) selectivity. Second, for approximately random
query-filter correlation, we assume that ACORN is preferable in the transition
region \(5\% \le s < 10\%\), while POST-FILTER becomes preferable once
selectivity reaches \(10\%\).

Based on these observations, we define the following routing policy:

\[
\textsc{Plan}(q,\phi)=
\begin{cases}
\text{PRE-FILTER},
& s < 5\%, \\[6pt]

\text{ACORN},
& c=\text{negative} \land s \ge 5\%, \\[4pt]

\text{ACORN},
& c=\text{random} \land 5\% \le s < 10\%, \\[6pt]

\text{POST-FILTER},
& c=\text{positive} \land s \ge 5\%, \\[4pt]

\text{POST-FILTER},
& c=\text{random} \land s \ge 10\%.
\end{cases}
\]

Although intentionally simple, this planner demonstrates how lightweight correlation estimates can be integrated into execution-time routing decisions for filtered vector search.


\section{Test Set}

We evaluate the routing heuristic on a held-out workload of \(10{,}000\)
queries. The goal is to compare the adaptive planner against the three fixed
execution strategies introduced earlier: PRE-FILTER, POST-FILTER, and ACORN.

For each query, we generate an associated filter predicate as follows. First, we
sample a target selectivity uniformly from \([0.0001, 0.5]\), corresponding to
predicates that retain between \(0.01\%\) and \(50\%\) of the dataset. Next, we
sample a predicate center uniformly from the empirical attribute distribution.
We then construct an interval around this center whose cardinality is as close
as possible to the sampled target selectivity.

For each resulting query--predicate pair, we compute the exact filtered
top-\(100\) nearest neighbors. These exact results serve as ground truth for
the recall evaluation.

This procedure decouples query vectors from predicate generation. As a result,
the test workload covers a broad range of selectivities without being tied to a
particular query--filter correlation pattern or artificially favoring any single
execution strategy.

The adaptive planner operates in two stages. In the \textbf{routing} stage, it estimates the correlation regime of each query--predicate pair and assigns the query to one of the three execution strategies. In the \textbf{execution} stage, the selected strategy is run, and its output is compared against the exact ground truth.

\subsection{App reviews}\label{subsec:app-reviews}

The \texttt{app\_reviews} validation set provides a more realistic test case than the synthetic SIFT and GIST attributes, since its filters are derived from attributes of the original data rather than assigned independently.
The dataset is described in Section~\ref{subsec:fdroid}.
The exact validation metrics for these workloads are reported in Appendix~\ref{sec:appendix-validation-full}.

\paragraph{Overall result.}

Across both app-review attributes, all three metrics detect non-random attribute--embedding structure.
However, this signal does not translate into a consistently superior planner.
POST-FILTER remains the fastest baseline, PRE-FILTER achieves the highest recall, and ACORN provides the strongest fixed speed--recall tradeoff.
The adaptive planner is sometimes faster than ACORN, but usually loses too much recall to dominate it.

The main limitation is therefore not the absence of correlation signal.
Rather, the routing policy does not exploit the signal aggressively enough.
Many queries are still routed to POST-FILTER, even though ACORN often provides a better recall-efficiency tradeoff on these workloads.

\paragraph{Heuristic metric.}

Figure~\ref{fig:valid-set-heur} and Table~\ref{tab:validation-app-reviews-heuristic} show that the heuristic planner detects useful structure, but does not exploit it strongly enough.
On both attributes, the planner remains biased toward POST-FILTER.
This makes it competitive in runtime, but limits the recall improvement that the correlation signal could provide.

The effect is clearest for star rating, where the planner is only marginally slower than POST-FILTER but remains far below ACORN in recall.
For text length, the routing is more balanced and the planner recovers more recall, but still not enough to match ACORN's tradeoff.
Thus, the heuristic metric is informative, but the current routing rule underuses the stronger search strategy.

Table~\ref{tab:validation-app-reviews-tanh-diagnostics} shows that the tanh metric changes the classification balance.
As discussed in Section~\ref{par:shrink-toward-zero}, the tanh term pushes weak local/global imbalances toward zero.
This makes \(c_{\mathrm{tanh}}\) more conservative for weak correlations: if the queried range is only mildly overrepresented or underrepresented in the local cluster, the resulting score is reduced and may fall into the random class.
This explains the star-rating workload, where several queries that were positive under the heuristic metric become random under \(c_{\mathrm{tanh}}\).

For text length, however, some queries exhibit a much stronger local lift.
That means the queried range contains substantially more probability mass inside the assigned cluster than in the global distribution.
Even if the overall cluster distribution is still relatively close to the global distribution, the strong range-specific lift can be large enough to keep the tanh score above its lower null threshold.
These queries may be random under \(c_{\mathrm{heur}}\), where the lift only contributes its sign and the score mainly depends on the JS divergence, but positive under \(c_{\mathrm{tanh}}\), where the strength of the lift also contributes.

Thus, \(c_{\mathrm{tanh}}\) suppresses weak correlations by pushing them toward zero, but it can still classify queries as correlated when the queried range is strongly overrepresented in the local cluster.
