\chapter{Methodology}\label{chap:methodology}

This chapter describes the experimental methodology used in this thesis. 
The goal is therefore not to present results, but to make the experimental setup precise: which data is used, how vectors and metadata are represented, and how the compared search strategies are implemented in our evaluation.

\section{Datasets}\label{sec:dataset}

We will evaluate our method on both synthetic and real-world data, allowing us to analyze its behavior under controlled conditions as well as in realistic settings.

\subsection{Test set: \textsc{SIFT} with controlled attributes}
\label{subsec:sift1m}

As the main controlled test set, we use the widely adopted \textsc{SIFT1M} dataset, introduced in~\cite{jegou2010product} and also used in later filtered vector search work such as ACORN~\cite{acorn}.

\textsc{SIFT1M} contains \(10^6\) base vectors and \(10^4\) query vectors in \(\mathbb{R}^{128}\).
The dataset also provides ground-truth nearest neighbors under Euclidean distance, which makes it a standard benchmark for approximate nearest neighbor search.

Since \textsc{SIFT1M} does not include structured attributes, we augment the base vectors with synthetic attributes following the controlled construction described in Chapter~\ref{chap:dataset-construction}.

The SIEVE benchmark augments these vector datasets with synthetic numeric attributes.
For each dataset, two numeric attribute columns are sampled independently from a standard normal distribution, \(\mathcal{N}(0,1)\), using a fixed random seed, and each column is then sorted.
Thus, each base vector receives two numeric attribute values.

The query ranges are not sampled independently for each query.
Instead, SIEVE first creates a fixed set of range predicates.
Each range predicate consists of fixed lower and upper endpoint values.
Queries are then assigned to these pre-generated ranges.
In our scalar first-attribute setup, this gives \(200\) unique range predicates over \(10{,}000\) queries; Appendix~\ref{sec:sieve-ranges} reports the ten most frequent and ten least frequent ranges.

The ranges are not used equally often.
A few ranges are assigned to many queries, while many others are assigned to only a small fraction of queries.
This type of uneven frequency pattern is commonly called Zipf-like: the most frequent items appear much more often than the less frequent ones.
For example, the most common range is used in \(10\%\) of all queries, whereas the least common ranges are used in only \(0.1\%\).
This means that the workload contains a few frequently repeated filters and many rarely used filters.

For \textsc{SIFT}, the query endpoints are sampled from \(\mathcal{N}(0,1)\), matching the base-attribute scale.
For \textsc{GIST}, the query endpoints are sampled from a narrower normal distribution, so endpoints are more concentrated near zero and usually define narrower range predicates.

Since this thesis considers only one filter predicate at a time, we project the SIEVE multi-attribute predicate to a scalar range workload by using only the first numeric attribute column.
For a query with first-column range endpoints \((a,b)\), the predicate is
\[
    a_{\min} \le x_{\mathrm{attr}} \le a_{\max},
    \qquad
    a_{\min}=\min(a,b),\quad a_{\max}=\max(a,b).
\]

The resulting selectivity distributions are shown in Table~\ref{tab:sieve-selectivity-summary}.
The two workloads differ substantially: \textsc{SIFT} covers a broad range of selectivities, including many medium- and high-selectivity queries, while \textsc{GIST} is concentrated closer to the low-selectivity regime.

\section{Implementation Details}\label{subsec:implementation-details}
For the execution strategies, expanding on what we dicussed in \ref{sec:execution-strategies} we will use the following implementations:
\paragraph{Pre-filter.}
The pre-filter baseline applies the metadata predicate before any vector ranking takes place. 
Given a query vector $q \in \mathbb{R}^d$ and a predicate interval $[l,h]$, the algorithm first identifies the subset of database vectors whose scalar metadata values satisfy the predicate. 
Formally, if the database is $\{(x_i, a_i)\}_{i=1}^N$, where $x_i \in \mathbb{R}^d$ is the vector embedding and $a_i \in \mathbb{R}$ is the associated metadata value, then the admissible set is
\[
X_{[l,h]} = \{x_i \mid l \le a_i \le h\}.
\]

To support efficient predicate evaluation, the metadata array is materialized offline in sorted order. 
More precisely, the implementation stores both a sorted metadata array and the corresponding permutation of vector identifiers. 
At query time, two binary searches are performed on the sorted metadata array to locate the left and right boundaries of the interval $[l,h]$. 
This yields a contiguous block of identifiers corresponding exactly to the admissible subset. 
Hence, predicate evaluation itself costs only $O(\log N)$ time.

Once this contiguous identifier range has been obtained, the implementation performs an exact scan over the corresponding vectors. 
For each admissible vector $x_i$, it computes the squared Euclidean distance
\[
d(q, x_i) = \|q - x_i\|_2^2
\]
by explicitly iterating over all $d$ dimensions. 
The resulting list of $(d(q,x_i), i)$ pairs is then partially sorted to retain only the $k$ smallest distances. 
If the admissible subset contains at most $k$ elements, the list is fully sorted; otherwise, a partial sort is used to avoid unnecessary ordering work beyond the top-$k$ frontier.

It is important to note that, in the current implementation, pre-filtering does \emph{not} perform ANN search on a dedicated filtered index, 
but computes exact distances to every vector in the filtered subset and then selects the exact top-$k$ within that subset. 
Algorithmically, this baseline is best understood as \emph{filter first, then exact $k$-NN on the filtered subset}.

\paragraph{Post-filter.}
The post-filter baseline first performs ANN search on the full vector dataset and applies the metadata predicate only afterwards.
The implementation uses a global HNSW index over all database vectors, so metadata is ignored during graph traversal and the returned ranking is based only on vector distance.

Given a query vector \(q\) and predicate interval \([l,h]\), the method retrieves a candidate list
\[
C_f(q) = \{x_{i_1}, x_{i_2}, \dots, x_{i_f}\},
\]
ordered approximately by increasing distance to \(q\).
It then scans this list in order and keeps candidates whose attribute satisfies
\[
l \le a_{i_j} \le h.
\]
The final answer consists of the first \(k\) admissible candidates found in this filtered ranking.

Since only a fraction \(s=m/N\) of the dataset satisfies the predicate, where \(m\) is the number of admissible vectors, the implementation over-fetches from HNSW.
The value of \(m\) is estimated by binary search over the sorted metadata array, as in the pre-filter case.
The initial fetch size is then set to
\[
f \approx \left\lceil \frac{k}{s} \cdot \alpha \right\rceil,
\]
where \(\alpha > 1\) is an over-fetching factor.
In the implementation, the default value is \(\alpha=2\).
Since \(k/s\) is only an average-case estimate, the extra factor reduces the chance of retrieving fewer than \(k\) admissible candidates.
The implementation also sets \texttt{efSearch} to at least \(f\), so that HNSW uses a large enough search budget for the requested candidate list.

If fewer than \(k\) admissible candidates are found after filtering, the fetch size is doubled and the HNSW search is repeated.
This retry process continues until either enough valid results are found, or a maximum retry or fetch budget is reached.
Thus, post-filtering is simple to implement, but its cost grows when the predicate is selective: as \(s\) decreases, the expected number of candidates that must be retrieved grows roughly as \(k/s\).

\paragraph{ACORN.}
ACORN is a hybrid vector search method designed to reduce the weaknesses of conventional post-filtering under structured predicates.
The implementation used in this work relies on the open-source ACORN library.\footnote{\url{https://github.com/guestrin-lab/ACORN}}

ACORN builds on HNSW, but modifies the graph so that it remains useful even when a query predicate removes many nodes.
In a standard HNSW index, each node stores a small set of neighbors chosen for efficient navigation in the full vector space.
This works well for ordinary ANN search, but it can break down under filtering.
If only a subset of nodes satisfies the predicate, the subgraph induced by those admissible nodes may be sparse or poorly connected.
As a result, a normal HNSW traversal may spend effort near invalid nodes or fail to reach enough valid nearest neighbors.

The main idea of ACORN is therefore to make the graph more robust before queries arrive.
During construction, ACORN stores a larger neighborhood for each node than standard HNSW.
If \(M\) is the usual target degree, ACORN-\(\gamma\) considers roughly \(M\gamma\) candidate neighbors.
This increases the chance that, for many possible predicates, each valid node still has enough valid neighbors after filtering.
To limit memory growth, ACORN then applies a pruning or compression step controlled by parameters such as \(M_\beta\).

At runtime, ACORN receives a query vector \(q\) and a predicate \(\phi\).
The search still starts from the global graph entry point, as in HNSW.
However, when ACORN expands a node, it does not simply use the ordinary neighbor list.
Instead, it filters the neighbor list according to \(\phi\) and continues the greedy search mainly through neighbors that satisfy the predicate.
In other words, the predicate is used during graph traversal, not only after retrieving candidates.
The goal is to approximate a search over the subgraph of all admissible nodes,
\[
X_\phi = \{x_i : \phi(x_i) = \mathrm{true}\},
\]
without explicitly building a separate HNSW index for \(X_\phi\).

This distinction is important.
ACORN does not first enumerate all admissible vectors, as pre-filtering does.
It also does not first retrieve an unfiltered top-\(k'\) list and then discard invalid results, as post-filtering does.
Instead, it tries to navigate directly through the predicate-satisfying part of a pre-built augmented graph.
This makes ACORN an inline or predicate-aware graph traversal method.
